## Layer 2 — Alternatives (17LIVE migration approaches)

Four architectural approaches considered:

### Alternative 1: Big-bang rewrite
Strip all feature work, rebuild entire system from scratch, flip when complete.

**Ruled out by:** Constraint 3 (Team capacity). You had 2 senior + 1 junior engineer. Leadership made deliberate tradeoff to prioritize feature velocity over migration speed. A 5-10 engineer dedicated team was not available.

### Alternative 2: Parallel run
Build entire new system in parallel with monolith. Validate new system produces identical output to old. Run both for period of time, then flip completely.

**Ruled out by:** Constraint 3 (Team capacity). Parallel run requires building the full new system upfront before any value is delivered. With only 3 engineers, you couldn't afford the upfront effort or the ongoing cost of running both systems in parallel.

### Alternative 3: Branch-by-abstraction
Add feature flags and abstraction logic to monolith. Implement same functionality again in microservices. Use flags to gradually switch traffic between old (monolith) and new (microservices) implementations.

**Ruled out by:** Constraint 3 (Team capacity). This approach duplicates work — you write every piece of logic twice (once in monolith with flags, once in microservices). With only 3 engineers, you cannot afford redundant implementation. Engineering effort must move you forward, not duplicate code.

### Alternative 4: Strangler fig (what you chose)
Gradually extract individual services from monolith one at a time. New services run in parallel with monolith. API gateway routes to old or new depending on what's been migrated. Feature by feature, migrate to microservices, validate, then abandon legacy code. Incremental value delivery throughout.

**Passes all constraints:** Constraint 3 (feasible with 3 engineers), Constraint 1 (low-risk rollout), Constraint 2 (features keep shipping), Constraint 4 (no stack replacement), Constraint 5 (POC unlocks SRE support).

## Layer 3 — Tradeoffs (what each alternative optimizes vs. sacrifices)

### Big-bang rewrite
**Optimizes for:**
- Speed (theoretically fastest path to complete migration if execution is perfect)
- Architectural cleanliness (fresh design from scratch, no legacy constraints)

**Sacrifices:**
- Risk mitigation (all-or-nothing flip means one outage breaks everything)
- Ability to course-correct (committed to design before validating it works at scale)
- Incremental value delivery (zero value until entire migration complete)
- User safety (one mistake could take down the entire platform across multiple regions during peak)

### Parallel run
**Optimizes for:**
- Complete validation before flip (you know new system works identically to old)
- Safety (you can run both in parallel indefinitely before committing to flip)

**Sacrifices:**
- Engineering efficiency (waste of capacity running duplicate systems)
- Speed (requires building entire new system before any value is delivered)
- Iterative learning (can't course-correct, already committed to building everything)

### Branch-by-abstraction
**Optimizes for:**
- Gradual rollout with feature flags (can roll back instantly if issues arise)
- Risk mitigation (no hard flip, just toggle flags)

**Sacrifices:**
- Code simplicity (feature flags and abstraction logic in monolith adds complexity)
- Engineering efficiency (duplicate logic in two places)
- Performance (feature flags and abstraction layers add runtime overhead)

### Strangler fig (what you chose)
**Optimizes for:**
- Risk mitigation (gradual rollout service-by-service means you catch problems early)
- Incremental validation (each service validated before moving to next)
- Ability to course-correct (you learn from each migration and adjust approach)
- Continuous value delivery (each migrated service delivers value immediately)
- Operational safety (if one service has issues, you roll back one service, not entire platform)

**Sacrifices:**
- Speed (slower overall than big-bang, takes longer to complete full migration)
- Architectural simplicity (managing monolith + microservices in parallel is complex)
- Operational overhead (running two systems means more monitoring, more moving parts)

## Layer 4 — Counterfactuals (what would change the decision)

Three scenarios where you would have chosen a different alternative:

### Counterfactual 1: Team capacity constraint changes
**If:** You had 5-10 engineers dedicated to migration work (Constraint 3 changes)
**Then:** Big-bang rewrite becomes viable. You could pause features temporarily, focus entire team on rewrite, and flip faster. Risk remains high, but it becomes a feasible tradeoff if speed is critical.

### Counterfactual 2: Feature velocity constraint changes
**If:** Feature velocity were less critical during seasonal events (Constraint 2 changes) — if you could pause major feature releases for 3–6 months
**Then:** Big-bang rewrite becomes more attractive. You could tell product: "we're pausing new features for the migration" and focus the org entirely on rewrite. You're no longer sacrificing revenue.

### Counterfactual 3: Scale/risk constraint changes
**If:** DAU were lower during peak Asia-evening windows, or you could tolerate brief outages (Constraint 1 changes)
**Then:** You have more tolerance for mistakes during migration. You could roll out faster, take more risks, use techniques like hot-swap deployments or dual-write periods that carry more risk but move faster. You'd sacrifice safety for speed.

**Key insight:** Each constraint removed changes which alternative becomes optimal. As long as all five constraints are in place, strangler fig is the right answer.
